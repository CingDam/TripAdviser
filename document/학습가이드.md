## 1. useCallback - 언제 써야 하고 언제 쓰면 안 되나

### 핵심 개념: 함수도 객체다

```typescript
// 컴포넌트가 렌더링될 때마다 이 함수는 새로 만들어짐
const handleClick = () => { console.log('click') };
// 매 렌더링마다 handleClick의 메모리 주소가 달라짐
// {} === {} 가 false인 것처럼, 함수도 매번 다른 참조값
```

useCallback은 **함수의 참조값을 고정**시킴

```typescript
const handleClick = useCallback(() => {
  console.log('click');
}, []); // 의존성이 바뀌지 않으면 항상 같은 함수 참조 반환
```

---

### 이 프로젝트에서 쓴 이유

```typescript
// usePlaceSearch.ts
const search = useCallback(
  async (query, types, panTo) => { ... },
  [placeLib, map]
);
```

`search`가 useEffect 의존성 배열에 들어가기 때문에 중요해요.

```typescript
// MapContainer.tsx
useEffect(() => {
  searchRef.current = search;
}, [search]); // ← search 참조가 바뀔 때만 실행되길 원함
```

useCallback 없이 일반 함수로 쓰면:
```
렌더링 → search 새로 생성 → useEffect 실행 → 렌더링 → search 새로 생성 → ...
무한루프
```

---

### 언제 써야 하나 (3가지 경우)

```typescript
// ① useEffect 의존성에 함수가 들어갈 때
useEffect(() => { ... }, [search]); // search가 안정적이어야 함

// ② 자식 컴포넌트에 함수를 props로 내릴 때 (자식이 React.memo인 경우)
<PlaceCard onAdd={handleAdd} /> // handleAdd가 매번 새로 생기면 PlaceCard도 매번 리렌더링

// ③ 이 프로젝트처럼 ref에 저장하는 함수
const searchRef = useRef(search);
```

---

### 언제 쓰면 안 되나 (남발 금지)

```typescript
// ❌ 이런 건 useCallback 필요 없음
const handleClick = useCallback(() => {
  setCount(count + 1);
}, [count]); // count 바뀔 때마다 어차피 새로 만들어짐 → 의미 없음

// ✅ 그냥 이렇게 써도 됨
const handleClick = () => setCount(count + 1);
```

```
판단 기준:
이 함수가 useEffect 의존성에 들어가거나, props로 내려가거나, ref에 저장되나?
→ YES → useCallback
→ NO  → 그냥 일반 함수
```

## 2. useMemo - useCallback이랑 뭐가 다른가

```
useCallback → 함수 자체를 메모이제이션
useMemo     → 함수의 실행 결과값을 메모이제이션
```

```typescript
// useCallback
const fn = useCallback(() => expensiveCalc(), [dep]);
// fn은 메모된 함수. 호출해야 결과가 나옴.

// useMemo
const result = useMemo(() => expensiveCalc(), [dep]);
// result는 이미 실행된 결과값. 바로 쓸 수 있음.
```

---

### 이 프로젝트에서 쓸 수 있는 예시

```typescript
// PlanContainer.tsx
// 현재 - 렌더링마다 매번 계산
const currentPlaces = isAllView
  ? dayPlans.flatMap((d) => d.places)
  : dayPlans.find((d) => d.date === selectedDate)?.places ?? [];

// useMemo 적용 - dayPlans나 selectedDate 바뀔 때만 재계산
const currentPlaces = useMemo(() =>
  isAllView
    ? dayPlans.flatMap((d) => d.places)
    : dayPlans.find((d) => d.date === selectedDate)?.places ?? [],
  [isAllView, dayPlans, selectedDate]
);
```

---

### useMemo 언제 쓰나

```typescript
// ① 무거운 계산 (배열 정렬, 필터링, 변환)
const sorted = useMemo(() =>
  [...places].sort((a, b) => b.rating - a.rating),
  [places]
);

// ② 객체/배열을 useEffect 의존성에 넣을 때
const options = useMemo(() => ({ zoom: 15, center }), [center]);
useEffect(() => { map.setOptions(options) }, [options]); // 참조 안정화

// ③ 자식 컴포넌트에 객체를 props로 내릴 때
const style = useMemo(() => ({ color: 'red', size: 14 }), []);
```

```
판단 기준:
렌더링마다 이 계산을 다시 해도 괜찮은가?
→ YES → 그냥 일반 변수
→ NO (무겁거나, 의존성 문제 있거나) → useMemo
```

## 3. useCallback vs useMemo 판단 기준

```
반환값이 함수인가?   → useCallback
반환값이 값인가?     → useMemo
```

```typescript
// 헷갈리는 경우: useMemo로 useCallback 흉내 가능
const fn1 = useCallback(() => doSomething(), [dep]);
const fn2 = useMemo(() => () => doSomething(), [dep]); // 동일하게 동작

// 그냥 용도에 맞게 쓰는 게 읽기 좋음
```

### 이 프로젝트 기준 정리

| 위치 | 현재 | 최적화 시 |
|------|------|----------|
| `usePlaceSearch.ts` - `search` | `useCallback` ✅ | 이미 올바름 |
| `PlanContainer.tsx` - `currentPlaces` | 일반 변수 | `useMemo` 고려 가능 |
| `SearchContainer.tsx` - `handleSearch` | 일반 함수 | 그냥 두면 됨 |

> **중요:** 최적화는 느려질 때 하는 거예요. 지금 프로젝트 규모에서 useMemo 없다고 문제 없어요. 개념만 알고 있으면 됨.

## 4. FastAPI 생태계 전체 그림

Spring Boot 쓸 때 내장된 것들이 FastAPI에서는 별도 패키지로 분리되어 있어요.

```
Spring Boot                   FastAPI 생태계
────────────────────────────────────────────────────
내장 서버 (Tomcat)            uvicorn (ASGI 서버)
Bean Validation (@Valid)      Pydantic
application.properties        pydantic-settings + .env
Spring Security CORS          starlette CORSMiddleware
Spring Data JPA               SQLAlchemy (ORM, 이 프로젝트엔 없음)
@Scheduled                    APScheduler / Celery
Actuator (헬스체크)            /docs 자동 생성 (Swagger 내장)
```

**이 프로젝트에서 실제로 쓴 것**
```
FastAPI       → 웹 프레임워크 (라우팅, 요청/응답)
Pydantic      → 데이터 검증 + 직렬화
uvicorn       → 실제로 HTTP 요청 받는 서버
pydantic-settings → .env 파일 읽기
LangChain     → Gemini API 호출 추상화
```

## 5. Pydantic - 데이터 검증의 핵심

```python
# core/models.py
class Place(BaseModel):
    place_id: str
    name: str
    rating: float | None = None  # 기본값 있으면 optional
```

### FastAPI가 자동으로 해주는 것

```python
@router.post("/sort", response_model=SortResponse)
async def sort(req: SortRequest):  # ← 여기서 마법이 일어남
    ...
```

```
클라이언트가 JSON 보냄
    ↓
FastAPI가 SortRequest 모델로 파싱 시도
    ↓
타입 불일치 → 422 Unprocessable Entity 자동 반환
타입 일치  → req 객체로 사용 가능
    ↓
SortResponse 반환 → FastAPI가 JSON으로 직렬화
```

### response_model의 역할

```python
@router.post("/sort", response_model=SortResponse)
# 1. 응답 데이터를 SortResponse 형태로 필터링
#    (SortResponse에 없는 필드는 자동으로 제거됨 → 데이터 노출 방지)
# 2. /docs (Swagger UI)에서 응답 스키마 자동 문서화
```

### pydantic-settings

```python
# config.py
class Settings(BaseSettings):
    gemini_api_key: str

    class Config:
        env_file = ".env"

settings = Settings()
# → .env 파일에서 GEMINI_API_KEY 자동으로 읽어옴
# → 없으면 앱 시작 시 ValidationError 발생 (런타임 중간에 터지는 것 방지)
```

## 6. uvicorn / ASGI - FastAPI가 실행되는 방식

```python
# main.py
if __name__ == "__main__":
    uvicorn.run("main:app", host="0.0.0.0", port=8000, reload=True)
```

```
Spring Boot: 내장 Tomcat (WSGI 아닌 서블릿 컨테이너)
FastAPI:     uvicorn (ASGI 서버)
```

**ASGI가 뭔가**

```
WSGI (구형) → 요청 하나 처리하는 동안 다른 요청 대기
ASGI (신형) → async/await로 여러 요청 동시 처리 가능
```

FastAPI의 `async def`가 의미있는 이유가 이것 때문이에요.

```python
# sync 함수 - uvicorn이 별도 스레드에서 실행
def sort(req: SortRequest):
    result = call_gemini(req)  # 이 동안 다른 요청 대기
    return result

# async 함수 - Gemini API 기다리는 동안 다른 요청 처리 가능
async def sort(req: SortRequest):
    result = await call_gemini(req)  # 기다리는 동안 다른 요청 처리
    return result
```

`reload=True` → 코드 수정하면 서버 자동 재시작 (개발용, 배포 시 제거)

## 7. LangChain - 이 프로젝트에서 쓰인 것만

```python
from langchain_google_genai import ChatGoogleGenerativeAI
from langchain_core.prompts import ChatPromptTemplate
```

### 왜 직접 Gemini API 호출 안 하고 LangChain 쓰나

```python
# Gemini SDK 직접 쓰면
import google.generativeai as genai
model = genai.GenerativeModel('gemini-2.5-flash')
response = model.generate_content(f"날짜: {date}\n장소: {places}")
result = response.text

# LangChain 쓰면
chain = sort_prompt | llm
response = chain.invoke({"date": date, "places": places})
```

직접 써도 되는데 LangChain이 주는 것:
- **프롬프트 템플릿 관리** → `{변수}` 치환 자동화
- **모델 교체 용이** → Gemini → GPT-4 바꿀 때 `llm =` 한 줄만 수정
- **체인 조합** → `prompt | llm | parser` 파이프라인 구성 가능

### 파이프 연산자 `|`

```python
chain = sort_prompt | llm
# sort_prompt의 출력(완성된 프롬프트 문자열)이 llm의 입력으로 들어감

# 더 길어지면
chain = prompt | llm | output_parser
# prompt 출력 → llm 입력 → llm 출력 → parser 입력
```

### temperature=0 의미

```python
llm = ChatGoogleGenerativeAI(
    model="gemini-2.5-flash",
    temperature=0,  # ← 이게 뭔가
)
```

```
temperature 0    → 항상 가장 확률 높은 토큰 선택 → 결과가 일정함
temperature 1    → 랜덤성 추가 → 창의적이지만 결과가 매번 다름
temperature 0.7  → 중간 (일반 대화에 흔히 씀)
```

일정 반환을 기대하는 JSON 파싱 작업이라 `0` 사용

## 9. TypeScript - 이 프로젝트에서 헷갈리는 문법만

Java 알면 타입 개념 자체는 낯설지 않을 거예요. 문법 차이만 짚어요.

```typescript
// Java라면: Integer rating = null;
// TS는:
rating?: number | null;
// ? → 아예 없어도 됨 (undefined)
// | null → 있는데 null인 것도 허용
// 둘 다 쓰는 건 "없거나, 있어도 null일 수 있음"
```

```typescript
// Java의 Optional.orElse() 같은 것들
rating: p.rating ?? null      // null/undefined면 null로 대체
name: p.displayName ?? ''     // null/undefined면 빈 문자열로 대체
lat: p.location?.lat() || 0  // location이 없으면 undefined → 0으로 대체
```

```typescript
// 타입 가드 - 이게 가장 낯선 부분
.filter((r): r is PromiseFulfilledResult<...> => r.status === 'fulfilled')
//      ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
// "이 조건이 true면 r의 타입을 확정해줘"
// 이게 없으면 filter 이후에도 TS가 타입을 모름
```

```typescript
// Record - Java의 Map<K, V>랑 같은 개념
const SEARCH_QUERIES: Record<SearchType, string> = {
  tourist: '관광지 명소',
  // ...
}
// SearchType 키만 허용, 값은 string
```

> **Java 경험자 관점:** `as string`은 Java의 강제 캐스팅이랑 비슷해요. 남발하면 런타임 에러 못 잡음. 이 프로젝트에서 `as string` 보이면 대부분 Gemini API 응답 타입이 불확실해서 임시로 쓴 것들이에요.

## 2. useEffect 의존성 배열 - 버그의 원인 1위

React 경험이 있어도 이 부분은 대부분 헷갈려해요.

```typescript
// MapContainer.tsx - MapHandler 안에 useEffect가 3개인 이유

// ① map, placeLib 둘 다 준비됐을 때 1번만 실행
useEffect(() => {
  searchRef.current('', ['tourist', 'restaurant', 'cafe'], false);
}, [map, placeLib]);

// ② 검색어가 바뀔 때마다 실행
useEffect(() => {
  searchRef.current(searchParams, ['tourist', 'restaurant', 'cafe'], true);
}, [searchParams]);

// ③ 지도 이동 완료될 때마다 실행
useEffect(() => {
  const listener = map.addListener('idle', () => { ... });
  return () => listener.remove(); // ← 클린업: 다음 실행 전에 이전 리스너 제거
}, [map]);
```

**의존성 배열 3가지 패턴**
```
[]             → 마운트 시 1번만
[a, b]         → a 또는 b가 바뀔 때마다
없음            → 매 렌더링마다 (거의 안 씀, 무한루프 위험)
```

**흔한 실수**
```typescript
// ❌ 의존성 빠뜨리기
useEffect(() => {
  search(searchParams); // searchParams 쓰는데
}, []);                 // 의존성에 없음 → 초기값만 캡처, 업데이트 안 됨

// ❌ 객체/배열을 의존성에 넣기
useEffect(() => {
  ...
}, [options]); // options가 객체면 렌더링마다 새 참조 → 무한루프
```

## 3. useRef vs useState - 이 프로젝트 핵심 패턴

```typescript
// MapContainer.tsx
const isPanning = useRef(false); // 왜 useState가 아닐까?
```

| | useState | useRef |
|--|---------|--------|
| 값 바뀌면 | 리렌더링 발생 | 리렌더링 없음 |
| 용도 | 화면에 보여야 하는 값 | 내부 플래그, DOM 참조 |

```typescript
// isPanning이 true/false 바뀐다고 화면을 다시 그릴 필요가 없음
// 그냥 "지금 지도 이동 중인가?" 체크용 플래그

isPanning.current = true;  // 검색 시작

map.addListener('idle', () => {
  if (isPanning.current) {
    isPanning.current = false;
    return; // 검색으로 인한 이동 → 재검색 스킵
  }
  search(...); // 사용자가 직접 움직인 경우만 재검색
});
```

**searchRef도 같은 이유**
```typescript

```

`idle` 리스너 콜백 안에서 `search`를 직접 쓰면 클로저 문제로 초기값만 캡처됨
→ ref로 감싸서 항상 최신 함수를 참조하게 만드는 패턴

## 8. Python - FastAPI가 Spring이랑 어떻게 다른가

Spring 알면 구조가 낯설지 않을 거예요. 대응 관계만 보면:

```
Spring                    FastAPI
──────────────────────────────────────────
@RestController          @router.get/post
@Service                 서비스 함수 (데코레이터 없음)
@RequestBody             Pydantic 모델 파라미터
@CrossOrigin             CORSMiddleware
application.properties   .env + pydantic_settings
@SpringBootApplication   app = FastAPI() + uvicorn
```

```python
# routers/sort.py
@router.post("/sort", response_model=SortResponse)
async def sort(req: SortRequest) -> SortResponse:
    return await sort_places(req)
# Spring의 @PostMapping("/sort") + @RequestBody 조합과 동일
```

**폴더 구조 분리 이유**
```
ai-server/
├── main.py          → Spring의 Application.java
├── routers/sort.py  → Spring의 Controller
├── services/sort_service.py → Spring의 Service
└── core/            → Spring의 domain/dto
```
Spring에서 Controller에 비즈니스 로직 안 넣는 것처럼, FastAPI도 라우터에 로직 안 넣음
